## 🚀 次回以降の同期手順

### Step 1: ローカルでプログラムの準備
新しいイベント用のフォルダを作成し、プログラムを用意します。
最低限以下の3つのファイルが必要です。
1.  **`app.py`** (LINE連携の窓口。Geminiへのプロンプトなどもここに記述)
    *   **⚠️【重要】** イベントごとに素材のアイコンが変わるため、`app.py` 内にある画像認識のプロンプト部分（`- copper (銅素材): 【ここを書き換える】` の箇所）を必ず、**実際のイベントアイテムの見た目（例：緑の野菜、赤い本など）**に書き換えてください。配置（左端など）で指定すると誤認識の原因になります。イベントアイテムの見た目がユーザー側からの資料になければ、ユーザーに送ってもらい、それを基に書き換えてください。
2.  **`calc_efficiency.py`** (周回計算の本体。**必ず `run_for_line()` というLINE返信用の関数を定義すること**)
3.  **`config.yaml`** (アイテムデータや現在の所持数などを定義)

### Step 2: `app.py` へのAPIキーの書き込み
PythonAnywhere上で環境変数が消失するのを防ぐため、`app.py` 内に直接APIキーを文字列として記述します。
（`os.environ[]` は使わず、シンプルにクォーテーション `' '` で囲んで代入します）

```python
# --- APIキーの設定 ---
LINE_ACCESS_TOKEN = 'LINE Developersで取得した長〜いアクセストークン'
LINE_CHANNEL_SECRET = 'LINE Developersで取得したチャネルシークレット'
GEMINI_API_KEY = 'Google AI Studioで取得したAPIキー'
```

### Step 3: PythonAnywhereへのアップロード
1.  [PythonAnywhereのFiles画面](https://www.pythonanywhere.com/user/irishu/files/home/irishu) を開きます。
2.  ローカルで準備した `app.py`, `calc_efficiency.py`, `config.yaml` の3つのファイルをアップロード（上書き保存）します。

### Step 4: サーバーの再起動（リロード）と有効期限の確認
1.  PythonAnywhereの画面右上から **[Web]** タブに移動します。
2.  （※重要）もし `This site has expired` と表示されている場合は、黄色の **[Run until 1 month from today]** ボタンを押してサーバーを起こしてください。
3.  一番上にある緑色の **[Reload irishu.pythonanywhere.com]** ボタンをクリックして、アップロードした最新のコードを反映させます。

### Step 5: 連携テスト
1.  [LINE Developers画面](https://developers.line.biz/console/channel/2009262603/messaging-api) を開きます。
2.  「Webhook設定」の項目にある **[検証]** ボタンをクリックします。
3.  **「成功」** と表示されれば連携完了です！
4.  実際にLINEアプリから「FGOイベント周回計算機」宛にスクショを送信し、正しい計算結果が返ってくるかテストしてください。

## ⚠️ トラブルシューティング

**Q. LINE Developersの「検証」を押すと 500 Internal Server Error が出る**
サーバーの裏側でプログラムがクラッシュしています。
PythonAnywhereの [Web] タブを下へスクロールし、`Log files:` の項目にある **`irishu.pythonanywhere.com.error.log`** を開いて一番下のエラーメッセージを確認してください。

*   **よくある原因1: `KeyError` や `SyntaxError`**
    *   `app.py` に記述したAPIキーのクォーテーション `' '` が抜けていたり、誤って `os.environ['キー']` と書いてしまっている可能性があります。
*   **よくある原因2: `ImportError: cannot import name 'run_for_line'`**
    *   新しく作った `calc_efficiency.py` の中に、LINE用の結果文字列を生成するための `run_for_line()` 関数を移植し忘れています。

## 📝 今回（Event 4）のシミュレーター構築プロセスとまとめ

今回（Event 4: 雀のお宿の活動日誌～閻魔亭繁盛記～）のシミュレーター構築では、以下のプロセスを経て最適化を行いました。次回以降の新しいイベント用スクリプトを作成する際の参考にしてください。

### 1. このチャット内で行ったことのまとめ
*   **イベント仕様の解析**: 提供された攻略記事（PDF）とスクリーンショットから、今回は「材木」という特殊な副産物があること、および礼装だけでなく「サーヴァントのクラスや特効」によるドロップボーナスがあることを特定しました。
*   **レイアウトの整理**: LINEボットから返ってくる `result.md` の出力を、絵文字（🟡⚪️🟠）やGitHub風アラート（`> [!TIP]`など）を用いて直感的に見やすく整形しました。
*   **LP（線形計画法）アルゴリズムの導入**: 「ある素材が目標に達するまで一直線に走る（貪欲法）」という従来のバグ・最適化不足を解消し、「複数の素材を同時にバランスよく集め切るための完璧なフェーズ切り替え（例: 最後に銅フレンドを2周だけ借りる等）」を自動で導き出し、手計算以上の「真の最短周回数（90周など）」を弾き出すシステムに進化しました。
*   **動的な副産物(木材等)計算・おすすめ判定**: 最も周回数の少ないプランを自動判別し `✨(おすすめ)` を付与する機能や、指定されたサーヴァント編成に応じた副産物の最終予測獲得数を動的に自動計算する機能を追加しました。
*   **複数クエストの同時シミュレーション**: `config.yaml` の `quests` ブロックにより、90++や90+など複数パターンのクエスト効率を同時にシミュレーションし、比較出力できるよう進化しました。
*   **簡潔版リザルトの自動生成**: 周回時にパッと見で編成を組めるよう、詳細な説明を省きトップ2のプランのみを抽出した `result_short.md` を `result.md` と同時に自動生成する機能を追加しました。

### 2. 今回用意した元データとシミュレーション作成の手順
今回のシステム構築にあたり、ユーザー側で以下のデータを用意し、AIに解析させました。

1.  **元データの投入**: 
    *   `元データ/攻略記事.pdf` : イベントの全体的な仕様（ドロップ枠数、特効サーヴァント、礼装ボーナスなど）を把握するための資料。
    *   `元データ/11.05.19.png` : イベント開始直後の画面スクショ。どの素材がどの程度必要か、またはドロップのUI構造を把握するための画像。
2.  **現状アイテムの投入**:
    *   `現在アイテム所持数/` フォルダに定期的にスクショをアップロードし、AIに現在の所持数（雀の幸・川の幸・山の幸、および木材）を割り出させました。
3.  **計算スクリプトの更新・対話**:
    *   解析結果をもとに `config.yaml` （目標数・現在数・基礎枠数）を作成・更新。
    *   `calc_efficiency.py` を実行して、プランA（最適切り替え）、プランB（金固定）、プランC（自前フリー）を自動生成させ、不都合な点（フェーズが無駄に分かれている、計算が1周多い等）をAIに指示して修正させました。

### 💡 【重要】イベント形式の推測と注釈について
今後のイベント（Event 5, 6...）においても、同様の元データ（攻略記事のPDFやスクショ等）をAIに読み込ませて仕様を解析・推測させることになります。これまでのイベント形式の内訳は以下の通りです。

**【過去のイベント形式の内訳】**
*   **(a) イベント4**：素材を集めるだけの形式（※副産物として木材あり）
    *   純粋に「金・銀・銅」の交換素材を特定の目標数まで集める形式です。線形計画法（LP）等による「どのフレンドを何周借りるか」の計算が最も綺麗に機能します。
*   **(b) イベント3**：ミッションと素材の形式
    *   素材集めに加え、「特定エネミーを〇〇体倒す」などのミッションが並行して存在します。ミッション達成を優先したルート構築が必要です。
*   **(c) イベント1か2**：ポイントと素材の形式
    *   素材だけでなく「〇〇ポイントを300万まで稼ぐ」といった条件が加わります。ポイント最高効率クエストを優先して回る必要が出てくるため、次回AIに元データを渡す際、明確に「今回はポイント形式なので、ポイント到達も条件に入れて計算してほしい」と指示出し（プロンプト調整）を行う必要があります。

### 3. イベントフォルダ（例: `event4`）の基本構造と役割分担
シミュレーターを構築・運用するにあたり、フォルダ内のファイルやディレクトリは「ユーザー（人間）が用意するもの」と「AIが生成・更新するもの」に分かれます。次回以降のフォルダ作成の目安にしてください。

**📁 ユーザー側から提供していただくもの（インプット）**
*   **`元データ/` フォルダ**
    *   イベント開始直後にユーザーが投入する最も重要なインプットフォルダです。今回の「Event 4」では、以下の3種類のファイル群（計10ファイル）が提供されました。
    *   **【Event 4における元データの内訳と役割】**
        *   **1. `攻略記事.pdf` (攻略サイトを丸ごと保存したPDF)**
            *   **役割**: イベントの全体仕様（素材ドロップ形式か等）の把握や、攻略の流れをAIに俯瞰させるための全体資料です。
        *   **2. スクリーンショット群 (計8枚)**
            *   **役割**: これらは実際のゲーム画面ではなく、攻略記事（AppMedia等）の重要部分を個別に切り取った画像です。AIはここから正確な数値やボーナス仕様をOCRで読み取って `config.yaml` を組み立てます。
            *   `11.04.27.png`: ショップ交換における「金素材（雀の幸）」の交換対象アイテムと必要総数の一覧
            *   `11.04.29.png`: ショップ交換における「銀素材（川の幸）」の交換対象アイテムと必要総数の一覧
            *   `11.04.32.png`: ショップ交換における「銅素材（山の幸）」の交換対象アイテムと必要総数の一覧
            *   `11.04.57.png`: イベント概念礼装・ガチャ礼装の「ドロップ獲得量増加ボーナス」の解説
            *   `11.05.04.png`: 特定の「特効サーヴァント（全木材+1、全木材+2など）」の一覧表
            *   `11.05.08.png`: 「基本クラス」がどの木材ドロップを増やすかの対応表
            *   `11.05.11.png`: 「エクストラクラス」がどの木材ドロップを増やすかの対応表
            *   `11.05.19.png`: 「木材の獲得計算・仕組み」および施設改築システムの詳細な解説
        *   **3. `所持礼装.md` (ユーザーの所持礼装状況メモ)**
            *   **役割**: ユーザー様がご自身のイベント概念礼装の「所持枚数」と「凸（限界突破）状態」を記述したマークダウンファイルです。AIはこれをもとに、周回計算に使用できる「自前の装備リソース」を正確に設定します。
    *   次回以降のイベントでも、このように「攻略記事のPDF」＋「重要部分を切り抜いたスクショ」＋「手持ち礼装のリスト」をセットでここに入れることで、計算の前提となる `config.yaml` が完璧に生成されます。
*   **`現在アイテム所持数/` フォルダ**
    *   周回を進める中で所持数に変動があった際、ユーザーが最新のスクショを撮影して投入します。
    *   AIがこの画像から現在の所持数を読み取り、プランを更新するための情報源とします。
*   **`クエスト情報/` フォルダ**
    *   クエストごとの詳細なドロップ内容や枠数が判明した際、その情報（スクショ等）を投入していただく場所です。

**🤖 AI側が作成・更新・出力するもの（アウトプット＆システム）**
*   **`config.yaml` （設定ファイル）**
    *   AIが元データから解析した目標数やボーナス値を定義するファイルです。ユーザーが用意したスクショをもとに、AIが「現在所持数」を逐次上書き更新します（※手動で書き換えることも可能です）。
*   **`scripts/calc_efficiency.py` （計算エンジン）**
    *   AIが作成する最適化シミュレーターの心臓部です。素材ドロップ型かポイント型かなど、イベント形式に合わせてAIがロジックをコーディング・修正します。
*   **`scripts/app.py` （LINE連携用スクリプト）**
    *   LINE Bot上でシミュレーターを稼働・応答させるためのPythonファイルです。
*   **`summary.md` （イベント仕様の要約）**
    *   AIが最初に元データを読み込んだ際、人間向けに「このイベントはどういう仕様で、どのようなアプローチが必要か」をまとめて書き出すドキュメントです。
*   **`result.md` （最終結果 / 詳細版）**
    *   計算スクリプトが吐き出した最終的な最適周回プランの詳細です。
*   **`result_short.md` （最終結果 / 簡潔版）**
    *   `result.md` から長文の説明を省き、周回に必要な「何周するか」「どの礼装を組むか」「最終的な獲得数」のみをランキング形式で抽出したファイルです。LINE Botの返信などにも活用可能です。
